# Scheduled Runs Resolve Source System Config at Execution Time

Scheduled runs execute outside an incoming HTTP request, so they cannot inherit the request middleware binding for source system configuration. We will resolve and bind the latest Source System Configuration from the scheduled run's source id at each Scheduled Run Boundary, rather than snapshotting configuration when the job is created, so changes from the system features page apply to existing scheduled work on its next run. `CronManager` owns this binding because it is the shared boundary for job, heartbeat, and dream execution; `CronExecutor` and the agent runtime should consume the bound context, not decide whether the run is scheduled work. The source configuration binding covers the whole scheduled-run business execution segment, not just `runner.stream_query()`, so source-scoped runtime behavior remains consistent across execution, notifications, records, and future scheduled-run side effects. The source configuration service will be passed explicitly through the runtime container into `CronManager`, rather than read from FastAPI request state or a module global. The boundary first uses an explicit scheduled-run source id, then falls back to a source id decoded from the run's runtime scope; only if neither exists does it treat the run as legacy source-less work. If a legacy scheduled run has no source id, we will preserve existing default behavior, leave Source System Configuration unbound, and log a warning rather than inventing a default source or failing the run. If the source configuration service is not available in the runtime, the scheduled run continues with Source System Configuration unbound. If a source id and service are both present, configuration resolution follows `SourceSystemConfigService.resolve_config()` semantics: stale cache/default results may be bound, but an unavailable or invalid configuration error fails the scheduled run instead of silently bypassing source-scoped controls.
