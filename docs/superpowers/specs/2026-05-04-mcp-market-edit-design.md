# MCP 市场编辑能力设计

## 背景

当前 CoPaw 的 MCP 市场已具备以下能力：

- 浏览市场 MCP 列表
- 查看 MCP 详情
- 测试连接
- 分发到租户
- 删除市场条目
- 上传新的 MCP 连接器

与 `D:\workspace\CmbCoworkAgent-main` 对照后，发现参考项目在市场条目层面还提供了 `编辑` 能力。
CoPaw 当前缺少该能力，导致以下问题：

- 上传后如果 `中文名称`、`描述`、`使用指引`、`可见机构` 填写不准确，只能删除后重新上传
- MCP 市场的维护体验弱于参考项目

本设计只补充 **MCP 市场条目的元数据编辑能力**，不扩展为连接器配置编辑或版本管理能力。

## 目标

为 MCP 市场增加 `编辑` 能力，用于修改市场展示元数据：

- 中文名称
- 描述
- 使用指引
- 可见机构

要求：

- 保持现有 MCP 市场主语义不变
- 不引入连接器配置更新
- 不引入重新上传文件更新
- 不引入版本递增
- 不影响已分发用户

## 非目标

本次不做以下能力：

- 编辑 MCP 的 `config`
- 编辑 `transport`、`command`、`args`、`url`、`headers`、`env`
- 重新上传 JSON 文件进行更新
- 修改 `Client Key`
- 修改英文名称 `name`
- 通过编辑触发版本号变化
- 通过编辑触发重新分发
- 引入 `安装 / 更新安装 / 卸载` 这类本地消费动作

## 维护路径划分

MCP 市场条目的后续维护分为两条路径：

### 路径 A：编辑

用于修改市场展示信息，只允许修改：

- 中文名称
- 描述
- 使用指引
- 可见机构

该路径不允许修改连接器配置本体。

### 路径 B：重新上传覆盖

用于修改连接器内容本体，包括：

- JSON 文件内容
- transport
- command
- args
- url
- headers
- env

该路径继续通过 `上传连接器` 入口完成。
若上传内容命中已有 MCP 身份，则直接覆盖原市场条目，而不是创建新条目。

## 设计原则

### 1. 编辑仅限市场元数据

`编辑` 的职责是维护市场展示信息，而不是修改连接器配置本体。

### 1.1 编辑与上传覆盖职责分离

必须明确区分两类维护动作：

- `编辑`：改展示信息
- `上传连接器`：改连接器内容

不允许把连接器内容更新混入 `编辑` 弹窗。

### 2. 版本语义保持稳定

市场列表中的 `version` 继续表示连接器内容版本。
纯元数据编辑不构成连接器版本升级，因此不修改 `version`。

### 3. 不影响运行态与分发态

编辑只更新市场元数据，不更新 `config`，因此：

- 测试连接逻辑不变
- 分发内容不变
- 已分发用户不受影响

### 4. 对齐参考项目的入口位置，不照搬其更新语义

参考项目的 `编辑` 允许带文件更新并自动递增版本。
CoPaw 只对齐入口位置与可维护性，不沿用文件更新语义。

## UI 设计

### 列表页

MCP 列表卡片动作区调整为：

- 详情
- 分发
- 编辑
- 删除

其中：

- `编辑` 仅管理员可见
- `编辑` 放在 `分发` 与 `删除` 之间

### 详情页

MCP 详情页右上角动作区调整为：

- 分发
- 编辑
- 删除
- 测试连接

其中：

- `编辑` 仅管理员可见

### 编辑弹窗

新增 `MCPEditModal`，使用当前 MCP 市场弹窗体系，不复用上传弹窗的文件上传区。

弹窗字段如下：

- 英文名称
  - 只读
  - 展示 `mcp.name`
- 中文名称（可选）
- 描述（可选）
- 使用指引（可选）
- 可见机构

不展示以下字段：

- Client Key
- 文件上传区
- JSON 模板帮助区
- 传输方式
- command / args / url / headers / env

弹窗中增加说明文案：

- `仅支持修改展示信息；如需修改连接器配置，请使用“上传连接器”重新上传覆盖。`

弹窗底部按钮：

- 取消
- 保存

## 权限设计

当前版本采用最小权限规则：

- 只有管理员可见 `编辑` 按钮
- 后端接口同样必须做管理员校验

不采用参考项目的：

- `item.canDelete`
- `item.ip === localIp`

原因是 CoPaw 当前 MCP 市场是服务端市场模型，不适合继续依赖本地 `localStorage + ip` 做资源拥有权判断。

## 数据模型

允许更新的字段：

- `chinese_name`
- `description`
- `guidance`
- `bbk_ids`

禁止更新的字段：

- `item_id`
- `name`
- `client_key`
- `config`
- `version`
- `creator_id`
- `creator_name`
- `created_at`

更新时：

- 保留 `created_at`
- 刷新 `updated_at`

## 上传覆盖规则

`上传连接器` 入口除“新增”外，还承担“覆盖已有 MCP 条目”的能力。

### 覆盖命中规则

若上传内容命中已有 MCP 身份，则：

- 复用原市场条目
- 不新增第二条记录
- 覆盖原市场条目的连接器内容

当前覆盖命中规则明确为：

- 以 MCP 的稳定业务身份字段命中已有条目
- 当前实现口径按 `name / client_key` 对应的同一身份判断
- 若命中已有条目，则覆盖该条目并复用原 `item_id`

不采用以下方式作为覆盖命中依据：

- 上传请求中手工指定新的 `item_id`
- 仅按中文名称匹配
- 仅按描述或使用指引匹配

### 覆盖后的数据行为

覆盖成功后：

- `item_id` 保持不变
- `config` 被新上传内容替换
- `updated_at` 刷新
- `created_at` 保持不变

### 覆盖时的元数据处理

重新上传覆盖时，如果上传弹窗中同时填写了：

- 中文名称
- 描述
- 使用指引
- 可见机构

则这些字段以本次上传值为准，一并更新到市场条目。

## 版本规则

版本语义区分如下：

- 纯元数据编辑：
  - 不修改 `version`
- 重新上传覆盖连接器内容：
  - 允许更新 `version`

版本字段继续表示连接器内容版本，而不是市场文案版本。

## 接口设计

新增 MCP 市场元数据更新接口：

`PUT /market/mcp/{item_id}/metadata`

请求体：

```json
{
  "chinese_name": "可选",
  "description": "可选",
  "guidance": "可选",
  "bbk_ids": ["可选机构列表"]
}
```

返回值：

- 返回更新后的 MCP 详情对象

错误语义：

- 条目不存在：返回 404
- 无权限：返回 403
- 参数非法：返回 400

## 上传接口职责

上传接口继续承担两种语义：

- 新增新的 MCP 市场条目
- 覆盖已有 MCP 市场条目

上传接口允许：

- 提交新的 JSON 文件
- 同时更新中文名称、描述、使用指引、可见机构

上传接口不作为“元数据轻编辑接口”的替代。

## 前端交互流程

### 列表页编辑

```text
点击编辑
-> 设置 editingMCP
-> 打开 MCPEditModal
-> 回填当前元数据
```

### 详情页编辑

```text
点击编辑
-> editingMCP = selectedMCP
-> 打开 MCPEditModal
```

### 保存成功

```text
保存成功
-> 关闭弹窗
-> 刷新 MCP 列表
-> 如果当前仍处于该条目的详情态
   -> 重新拉取当前详情
```

### 上传覆盖成功

```text
上传成功
-> 若命中已有 MCP
   -> 覆盖原市场条目
-> 刷新 MCP 列表
-> 若当前处于该条目详情态
   -> 重新拉取当前详情
```

## 后端处理流程

```text
接收更新请求
-> 校验管理员权限
-> 根据 item_id 查询市场 MCP
-> 校验条目存在
-> 仅更新元数据字段
-> 保留 config / version / created_at
-> 刷新 updated_at
-> 返回更新后的详情对象
```

## 与现有功能的关系

### 与上传连接器的关系

- 上传连接器：创建新的市场条目或覆盖连接器内容
- 编辑：仅修改已有市场条目的元数据

两者职责不同，不复用同一交互链路。

### 与测试连接的关系

编辑不改 `config`，因此测试连接逻辑无需调整。

### 与分发的关系

编辑不改分发内容，分发逻辑无需调整。

重新上传覆盖也不自动影响已分发用户。
只有管理员后续再次分发，目标租户才会拿到新的连接器配置。

### 与删除的关系

删除仍保持现有语义：

- 直接删除市场条目
- 不影响已分发用户

- 新增 service 更新方法

## 风险与控制

### 风险 1：误把编辑做成配置编辑

控制方式：

- 弹窗中不展示配置字段
- 后端 schema 不接受 `config`

### 风险 2：元数据编辑引起版本语义混乱

控制方式：

- 设计上明确：元数据编辑不改 `version`

### 风险 2.1：用户混淆“编辑”和“重新上传”

控制方式：

- 编辑弹窗明确提示“仅支持修改展示信息”
- 上传弹窗提示“命中已有 MCP 时将覆盖原条目”

### 风险 3：仅前端做权限控制

控制方式：

- 后端同步做管理员校验

## 结论

CoPaw 的 MCP 市场应补充 `编辑` 能力，但应严格限制为：

- 仅编辑市场元数据
- 不编辑连接器配置
- 不重新上传文件
- 不改版本
- 仅管理员可编辑

同时，连接器内容更新继续通过 `上传连接器` 完成：

- 编辑：改展示信息
- 上传连接器：新增或覆盖连接器内容

这能在不引入额外复杂度的前提下，补齐 MCP 市场维护能力，并与参考项目形成可接受的一致性。
