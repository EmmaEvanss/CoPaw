# Business Overview 页面桌面端 UI 还原设计

## 文档目的

本文档定义 `BusinessOverview` 页面本轮 UI 改造的目标、范围、结构、视觉规范和实现边界。

本轮工作的核心目标不是重新设计业务总览，而是在保留现有数据联动、筛选能力和弹窗交互的前提下，按参考截图完成桌面端页面的高保真还原。

本文档覆盖并细化 `docs/superpowers/specs/2026-04-28-business-overview-ui-design.md` 在本次需求中的适用范围。若两份文档存在冲突，以本文档为准。

## 已确认约束

- 目标页面：`console/src/pages/Analytics/BusinessOverview`
- 目标端：桌面端优先
- 视觉目标：约 90% 还原参考截图
- 数据层要求：保留现有接口请求、数据联动和筛选逻辑
- 交互要求：保留现有用户弹窗、技能弹窗、榜单滚动加载等行为
- 允许范围：允许针对现有数据结构做少量展示层自适应调整
- 不在本轮范围：移动端精细适配、接口重构、统计口径重定义

## 设计目标

### 1. 视觉目标

页面整体观感应接近参考截图的运营驾驶舱风格：

- 浅色背景、白色卡片、轻描边、弱阴影
- 清晰的模块分区和一致的卡片体系
- 指标卡、趋势图、排行榜、环图和四宫格信息块组成稳定的视觉节奏
- 页面在常见桌面宽度下应接近截图中的相对位置、比例和层级关系

### 2. 体验目标

- 页面首屏应能快速识别关键指标和趋势变化
- 用户不需要改变既有使用方式即可继续完成筛选、查看榜单和弹窗下钻
- 当真实数据长度、数量与截图不完全一致时，布局仍然稳定可读

### 3. 工程目标

- 仅调整展示层，不破坏现有数据获取链路
- 将展示映射逻辑集中管理，减少散落在 JSX 中的条件分支
- 为后续继续微调视觉或替换局部图形样式保留清晰边界

## 页面信息结构

页面按参考截图重组为以下桌面端结构。

### 1. 顶部筛选栏

顶部为单行工具区：

- 左侧：时间粒度切换 + 日期区间
- 右侧：机构/平台筛选

该区域的职责是统一承载所有页面级筛选，不再让页面标题和筛选器视觉竞争。

### 2. 第一屏 KPI 区

第一屏为六张并列指标卡，每张卡固定包含四段内容：

- 圆形彩色图标
- 指标标题
- 主数值与环比
- Top5 明细横条

六张卡在视觉上统一模板，仅通过颜色、图标和数据映射体现差异。

### 3. 第二屏分析区

第二屏分三列：

- 左侧：调用量趋势
- 中间：活跃用户排行榜
- 右侧：使用深度四宫格

这一区域承担“趋势 + 排行 + 深度指标”的复合分析入口。

### 4. 第三屏执行区

第三屏分三列：

- 左侧：任务执行概览环图
- 中间：技能使用 TOP5
- 右侧：MCP 调用概览环图

该区域承担“执行结果、能力热点、服务调用质量”的信息汇总。

## 数据映射策略

本轮不以“截图字段和真实字段一一对应”为前提，而是以“截图结构稳定复现”为目标，对现有数据做展示层映射。

### 1. KPI 卡映射

KPI 卡统一抽象为以下展示模型：

- `title`
- `value`
- `change`
- `icon`
- `accentColor`
- `topBreakdown`

数据来源优先使用现有：

- `overviewStats`
- `growthStats`
- `channelDistribution`

若某项指标缺少截图中的同名字段，则使用最接近的现有统计项替代，但保持卡片结构不变。

`topBreakdown` 优先从平台/机构分布数据中取前五项，渲染为横向进度条和占比文本。

### 2. 趋势图映射

趋势图继续使用现有 `trendData`，但视觉形态改造成与截图接近的双序列组合图：

- 一组主序列使用蓝色柱状表现
- 一组辅助序列使用绿色折线表现

若三条序列全部同时呈现会影响截图还原度，则优先保留最符合截图表达的两条核心序列。

### 3. 用户榜单映射

现有 `callsUsers` 和 `activeUsers` 的数据源保留。

页面主榜单按截图优先展示一张“活跃用户排行榜”，榜单项结构统一为：

- 排名徽标
- 用户展示名
- 脱敏标识或辅助身份信息
- 指标值

点击行为继续沿用现有用户弹窗逻辑。

### 4. 使用深度四宫格映射

截图中的四个小指标块不要求字段完全同名。

该区域优先从现有统计中选择四个稳定可得指标：

- 会话深度类指标
- 多轮占比类指标
- 停留时长类指标
- 人均使用类指标

若某项无法直接获得，则允许通过已有统计计算或用最接近指标降级替代，但四宫格布局和视觉结构保持固定。

### 5. 任务执行概览映射

任务执行概览卡优先采用环图 + 右侧图例结构。

若当前数据没有完整的截图同款状态拆分，则使用现有可得状态聚合结果，并按“成功 / 失败 / 进行中”或最接近的三类展示。

### 6. 技能与 MCP 卡映射

现有 `skills` 和 `mcpServers` 继续作为主要数据来源。

- 技能卡采用横条排行结构
- MCP 卡采用环图 + 图例结构

技能项点击仍需打开现有技能详情弹窗。

## 视觉还原规范

### 1. 页面背景

- 页面背景使用浅灰白色，接近 `#f5f7fb`
- 整体避免重阴影、重渐变和过深边框
- 重点通过留白、卡片分组和色彩重音建立层级

### 2. 卡片规范

所有主卡片统一：

- 白色背景
- 约 `12px` 圆角
- 轻描边
- 弱阴影
- 统一内边距

当前页面中偏硬的 `4px` 圆角和过强边框观感应整体替换。

### 3. 顶部控件规范

- 时间粒度使用浅底 segmented control
- 日期选择器和下拉筛选器统一为白色面板样式
- 控件高度、圆角、字重和间距统一
- 保留 Ant Design 原生交互，不重写交互机制

### 4. KPI 卡规范

每张 KPI 卡统一为：

- 顶部左侧彩色圆底图标
- 标题和主值构成视觉中心
- 环比置于主值附近或其下方
- 下部为 Top5 横条列表

建议颜色分配接近参考截图：

- 蓝
- 绿
- 青
- 橙
- 紫
- 深蓝

### 5. 图表规范

趋势图：

- 蓝色柱状
- 绿色折线
- 浅灰网格
- 顶部左对齐图例

环图：

- 中心大数字
- 右侧图例说明
- 图例项含彩色圆点、分类名和占比

横条排行：

- 细圆角进度条
- 左侧标签
- 右侧数值
- 统一节奏，不出现拥挤堆叠

### 6. 排版规范

页面内采用三级排版层级：

- 页面级标题：最大最醒目
- 区块标题：中等字重
- 卡片标题：14px 左右

数值统一使用更强字重与等宽数字对齐，辅助说明使用浅灰色，避免所有文字争抢视觉重心。

## 自适应策略

本轮以桌面端为主，不追求截图在所有窄宽度上的完全一致。

### 桌面端优先策略

- 核心布局围绕常见桌面宽度稳定呈现
- 卡片区域优先固定高度或相对稳定高度
- 列表区域内部滚动，不让整页高度失控
- 文案和名称允许截断，不强行撑开布局

### 自适应容错

允许以下展示层自适应：

- 排行名称截断 + Tooltip
- 进度条长度随容器缩放
- 数值格式化为 `K / M / % / s`
- 图例行数随真实数据略有变化

不允许为了适应极端数据而破坏整体骨架比例。

## 组件组织建议

不新增重型公共组件，优先在页面内部做展示层抽象。

建议将页面整理为以下渲染单元：

- `filter toolbar`
- `kpi card config`
- `trend panel`
- `ranking panel`
- `depth stat panel`
- `donut summary panel`
- `top list panel`

这些单元可以继续保留在 `index.tsx` 中，也可以在当前目录下拆成轻量子组件，但不应牵动业务数据流。

## 风险与应对

### 1. 截图字段与真实字段不完全一致

应对方式：

- 优先保持结构一致
- 用最接近的现有字段表达
- 将映射集中到展示层，不污染请求逻辑

### 2. 数据长度导致布局抖动

应对方式：

- 截断长文案
- 固定卡片高度
- 将列表改为内部滚动
- 使用等宽数字减少抖动

### 3. 过度追求静态复刻影响真实可用性

应对方式：

- 允许 10% 的自适应偏差
- 优先保留真实交互可用性
- 保持筛选、弹窗、滚动加载原行为

## 验收标准

本轮完成后，应满足以下标准：

- 桌面端页面结构与参考截图接近 90%
- 顶部筛选区、六张指标卡、趋势区、活跃用户榜、使用深度区、任务概览区、技能区、MCP 区位置关系基本一致
- 原有筛选逻辑可继续使用
- 原有弹窗入口可继续使用
- 原有榜单滚动加载可继续使用
- 页面在真实数据下不明显崩坏

## 实施顺序

1. 先重组页面骨架与区块顺序
2. 再统一卡片体系与顶部筛选视觉
3. 再逐块替换趋势、榜单、四宫格、环图和横条样式
4. 最后处理真实数据下的截断、滚动和边界态
