## Context

`/api/agent/init` 的路由实现位于 `src/swe/app/routers/agent.py`，当前在完成请求校验后，直接调用 `AgentMdManager.append_working_md()` 将 `text` 以追加模式写入目标文件。`AgentMdManager` 目前不区分“内容是否已在文件尾部存在”，因此同一初始化请求多次触发时会重复落盘。

这个变更只涉及单个接口的写入语义，不引入新的外部依赖，也不需要调整租户隔离、鉴权或前端请求结构。

## Goals / Non-Goals

**Goals:**
- 让 `/api/agent/init` 对相同尾部内容具备幂等性。
- 保持现有请求格式、文件名校验和租户/agent 解析链路不变。
- 将“是否需要真实追加”的判断落在靠近文件写入的位置，避免未来同类调用绕过规则。
- 为新增语义提供可回归的单元测试。

**Non-Goals:**
- 不修改 `/api/agent/init` 以外的 Markdown 写入接口行为。
- 不引入全文去重、段落级去重或模糊匹配，仅比较文件末尾与请求 `text` 的精确一致性。
- 不变更接口请求参数，也不新增外部存储或缓存。

## Decisions

### 决策 1：在 `AgentMdManager` 内封装尾部去重追加

**选择**：将“文件末尾是否已匹配待追加内容”的判断放在 `AgentMdManager.append_working_md()` 中，而不是只放在路由层。

**理由**：
- 该规则本质上属于文件写入语义，放在 manager 层更贴近职责边界。
- 路由层可以继续只负责请求校验和工作区解析，避免业务细节下沉到接口代码中。
- 后续如果有其他调用点复用 `append_working_md()`，也能自动获得一致语义。

**备选方案**：
- 在 `append_init_text()` 路由函数中先读取文件、再决定是否调用 `append_working_md()`：实现也可行，但会把文件内容判断逻辑散落在接口层，不利于复用。

### 决策 2：使用“精确尾部匹配”作为跳过条件

**选择**：只有当现有文件内容 `endswith(content)` 为真时，才跳过本次追加。

**理由**：
- 用户需求明确要求“判断当前文件末尾是否和 append 的 content 一致”。
- 精确尾部匹配规则简单、可预测，不会因为空白折叠、换行归一化等额外处理引入意外行为。
- 该规则天然兼容“文件不存在”“文件长度短于待追加文本”“中间存在相同片段但尾部不同”等场景。

**备选方案**：
- 对换行进行归一化后再比较：可能掩盖真实内容差异，超出当前需求。
- 扫描全文后去重：会改变现有接口的语义边界，不符合最小变更原则。

### 决策 3：保持接口成功响应形态不变

**选择**：当检测到尾部已一致而跳过写入时，接口仍返回 200，并保留现有响应结构。

**理由**：
- 本次变更目标是让初始化写入变为幂等 no-op，而不是引入新的调用分支。
- 现有调用方已将 200 结果视为初始化流程成功，维持成功响应可以避免连带调整前端初始化链路。

**备选方案**：
- 通过新增字段或修改 `appended` 语义暴露“本次是否实际写入”：可观察性更强，但会扩大接口契约变化范围，不是当前需求的最小实现。

## Risks / Trade-offs

- [风险] `append_working_md()` 的语义从“无条件追加”变为“幂等追加”，若其他调用方依赖重复写入，行为会变更。
  缓解：当前代码内仅发现 `/api/agent/init` 直接调用该方法，变更范围可控。
- [风险] 精确比较不处理换行归一化，调用方若传入不同换行格式，仍会被视为不同内容并追加。
  缓解：这与当前需求一致，并避免引入隐式内容改写。
- [风险] 为比较尾部需要先读取已有文件内容，超大文件会多一次读取开销。
  缓解：该接口面向顶层初始化 Markdown，文件规模有限，额外成本可接受。
