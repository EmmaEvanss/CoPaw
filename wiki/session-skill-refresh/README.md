# 会话间技能刷新说明

本文档面向 Swe 的普通使用者和技能维护者，说明 `feat(runner): refresh session skills between turns` 这个特性会解决什么问题、在什么情况下生效，以及你可以预期到什么行为。

本文只讲使用语义，不展开内部实现细节。

## 一句话理解

如果你在同一个 chat session 里已经实际使用过某个技能，然后在两轮对话之间更新了这个技能，Swe 会在下一轮开始前重新按最新技能内容继续工作，而不是继续依赖上一轮形成的旧假设。

## 这个特性解决什么问题

在没有这个能力之前，技能通常只在“被启用并实际关联”的当下进入当前轮次。之后即使你修改了技能目录、替换了技能实现，或者让这个技能不再对当前轮有效，同一个 session 后续轮次仍可能沿用旧技能内容带来的理解。

这个特性补上了“跨轮次刷新”这一步：

- 已经在当前 session 中确认使用过的技能，会被持续跟踪。
- 下一轮开始前，系统会检查这些技能相对上一轮是否发生了有效变化。
- 如果发生变化，后续回答会以当前技能内容为准。

## 什么时候会生效

这个特性同时满足下面几个条件时才会生效：

1. 你处在同一个 chat session 中，而不是新开一个独立会话。
2. 某个技能在之前轮次里已经被真正使用过，而不只是“可能相关”。
3. 变化发生在两轮之间，而不是当前轮执行过程中。
4. 下一轮开始时，这个技能出现了下面至少一种情况：
   - 技能目录内容发生变化
   - 同名技能切换到了新的技能目录
   - 这个技能不再属于当前轮的有效技能集合

可以把它理解成：

- “本轮用过，下一轮前检查是否变了”
- 不是“所有启用技能都一直被监控”
- 也不是“技能文件一改，当前正在执行的这轮立即热更新”

## 哪些技能会被跟踪

系统只跟踪“在当前 session 里已经确认关联过”的技能。

这意味着：

- 明确触发并实际使用过的技能会进入跟踪集合。
- 只是被模型低置信度提到、但并没有真正激活的技能，不会进入跟踪集合。
- 没在当前 session 里实际用过的技能，即使磁盘上发生变化，也不会因为这个特性触发刷新。

## 用户会看到什么

大多数情况下，你不会看到一条专门展示给用户的“技能已刷新”聊天消息。

这个特性的重点是让后续轮次的模型上下文自动切到最新技能状态，所以更准确的理解是：

- 你看到的是后续回答行为变了
- 而不是界面里多了一条显式通知

如果技能已经更新，下一轮回答会基于新的技能内容继续。

如果技能已经撤回或当前轮不再有效，下一轮回答会停止依赖这个技能之前带来的假设。

## 典型使用场景

### 场景一：你在调试某个技能

例如你已经在一个 session 里使用了某个本地技能，随后修改了它的 `SKILL.md`、脚本或相关资源文件。

这时你不需要手动重建整个会话。只要在同一个 session 中继续发起下一轮请求，Swe 会在下一轮开始前按最新技能内容刷新。

### 场景二：同名技能切到了新的目录版本

例如某个技能原来指向旧目录，后来被替换为新的目录实现。

同一个 session 的下一轮会按新的目录内容工作，不会继续把旧目录当作当前有效版本。

### 场景三：某个技能当前轮已经不再适用

例如某个技能之前用过，但后来它不再属于当前来源、当前配置或当前有效集合。

下一轮开始时，系统会把它视为已撤回，后续回答不再继续依赖它。

## 不会做什么

为了避免误解，这个特性不承诺下面这些行为：

- 不做 mid-turn 热更新。当前轮执行到一半时改技能，不会立刻影响这一轮。
- 不扫描全部启用技能。只检查当前 session 里已经确认使用过的技能。
- 不把“可能相关但没真正激活”的技能加入跟踪。
- 不保证提供用户可见的刷新提示消息。
- 不保证对每一种极细粒度的文件变化都用同一种方式对外表现。

## 建议的使用方式

如果你经常一边对话一边调技能，推荐按下面的方式理解和使用：

1. 先在 session 中实际触发一次目标技能。
2. 在两轮对话之间修改技能内容。
3. 保持使用同一个 session，继续发送下一条消息。
4. 预期下一轮开始时系统会自动按新技能状态继续。

如果你希望验证某次技能修改是否已经被采纳，最直接的方法不是找系统提示，而是观察下一轮回答是否已经体现新的技能规则、说明或行为。

## 适用边界

这个特性最适合下面这类场景：

- 在长会话里迭代本地技能
- 调整技能提示词、资源文件或脚本后继续追问
- 希望同一个 session 能自然衔接到新的技能版本

如果你的目标是“当前这一轮执行到一半就立刻切换技能内容”，这不属于本特性的范围。
